## 一、首屏性能基线

### 1.1 资源加载分析

| 指标 | 当前值 | 目标值 (v2) | 差距 |
|------|--------|-------------|------|
| HTML 主文件 | ~14.6 KB | - | - |
| CSS 总大小 | ~3.2 KB (gzip前) | - | - |
| JS 总大小 | ~109 KB (dashboard.js) + 其他 | - | - |
| Chart.js CDN | 从 CDN 加载 (umd.min.js) | 按需加载 | 需要优化 |
| 首屏请求数 | 约 15+ 个 (CSS/JS/API) | <= 10 | -30% |

### 1.2 关键渲染路径问题

1. **同步加载 Chart.js**: index.html 第 10 行同步加载 Chart.js，阻塞首屏渲染
2. **大量内联样式**: dashboard.js 中大量动态生成 HTML 带内联样式
3. **无代码分割**: 所有 JS 一次性加载，没有按需加载机制
4. **无资源预加载**: 缺少关键资源的 preload/prefetch

### 1.3 当前架构瓶颈

```
index.html
├── Chart.js (同步 CDN 加载 - 阻塞)
├── style.css (同步)
├── layout-improvements.css (同步)
├── right-sidebar.css (同步)
└── dashboard.js (109KB - 包含所有逻辑)
    ├── 系统概览
    ├── Agent 列表
    ├── 任务列表
    ├── 模型统计
    ├── 技能统计
    ├── 日志面板
    └── 图表管理
```

## 二、实时刷新机制基线

### 2.1 当前刷新频率

| 数据源 | 刷新方式 | 频率 | 问题 |
|--------|----------|------|------|
| WebSocket | 推送 | 5秒 | 全量数据推送 |
| 轮询 (fallback) | HTTP GET | 10秒 | 全量数据拉取 |
| 日志更新 | 定时器 | 1秒节流 | 整页重渲染风险 |
| 图表更新 | 定时器 | 30秒 | 无批量合并 |

### 2.2 数据更新问题

1. **全量替换**: WebSocket 每次推送都是完整数据快照
2. **无增量机制**: 没有 since/cursor 参数支持
3. **无可见性检测**: 页面切后台后仍保持高频刷新
4. **多通道冲突**: WebSocket + 轮询可能同时工作

## 三、列表性能基线

### 3.1 Agent 列表

- 当前实现: 一次性渲染全部 Agent
- 无虚拟滚动
- 无分页/游标
- 子 Agent 递归渲染，层级深时性能下降

### 3.2 日志列表

- 当前实现: 固定 50 条，无流式追加
- 每次更新全量 innerHTML 替换
- 无最大容量限制
- 无滚动性能优化

### 3.3 任务列表

- 当前实现: 最多显示 10 条
- 无分页机制
- 搜索/筛选无防抖

## 四、图表性能基线

### 4.1 Chart.js 使用现状

- 4 个图表同时初始化
- 无按需加载
- 更新频率: 30秒一次全量更新
- 无节流/防抖

### 4.2 图表数据问题

- 每次请求全量历史数据
- 无时间窗口分层
- 无数据采样/聚合

## 五、后端接口基线

### 5.1 当前 API 结构

```
GET /api/dashboard              -> 返回所有数据 (大对象)
GET /api/agents/list            -> 返回完整 Agent 列表
GET /api/tasks/current          -> 返回当前任务
GET /api/tasks/history          -> 返回历史任务
GET /api/logs/recent?count=50   -> 返回最近日志
GET /api/models/usage           -> 返回模型使用统计
...
```

### 5.2 接口问题

1. **无摘要接口**: /api/dashboard 返回全量数据
2. **无增量接口**: 没有 since/cursor 支持
3. **无分页**: Agent/任务/日志都是全量返回
4. **无压缩**: 依赖 Express 默认 gzip
